# VimKeys v0.6 smoke test

Three deliverables: Esc-Esc suspend chord, bookmark vomnibar (`b` / `B`,
needs Full Disk Access), layout-aware keyboard resolution (everyone but
the engine team will only notice this if they use a non-US layout).

## Setup

1. Build Release, install to `/Applications`. TCC grants carry over.
2. Launch VimKeys; confirm Apple Events grant still active.

## V-M5 regression checks

3. Per-site disable via Settings → Sites still flips status to
   "Off (this site)" within ~1.5s of navigation.

## V-M6 (post-trio) — Esc-Esc suspend chord

4. Navigate Safari to a page you haven't disabled (e.g. apple.com).
   Status reads "Normal".
5. Press `Esc Esc` quickly (within ~350 ms). Status flips to
   "Off (this site)" — VimKeys is session-suspended on that exact URL.
6. Press `j`. The letter `j` types into the page; no scroll.
7. Press `Esc Esc` again on the same URL. Status returns to "Normal".
   `j` scrolls again.
8. Suspend the URL again with `Esc Esc`. Then navigate to a different
   URL. Status returns to "Normal" automatically (session suspend
   cleared on navigation).
9. Return to the originally-suspended URL. Status stays "Normal" —
   session suspend was cleared by the navigation away.

## Bookmark vomnibar (`b` / `B`)

10. Press `b`. The first time, the vomnibar appears but reads zero
    bookmarks; an error banner flashes "Grant Full Disk Access…".
11. Open System Settings → Privacy & Security → Full Disk
    Access. Click the lock to unlock. Drag VimKeys from /Applications
    into the list (or click `+` and pick it). Toggle on.
12. Use the menu-bar "Restart VimKeys" button so the next bookmark read
    happens with the new TCC grant.
13. Press `b`. Vomnibar now lists Safari's bookmarks. Type a fragment
    of any bookmark's title — list narrows. Press Enter — Safari
    navigates to that URL in the current tab.
14. Press `B`. Same UI; Enter opens in a new tab.
15. With a long bookmark list, the suggestion list caps at 200 rows so
    scrolling stays cheap.

## Layout-aware keyboard resolution

16. Open System Settings → Keyboard → Input Sources. Add
    "Dvorak" (or any non-QWERTY layout). Switch to it via the
    menu-bar input switcher.
17. In Safari, on a regular page, press the physical key labelled
    `j` on QWERTY (which under Dvorak produces `c`). Page should
    NOT scroll — VimKeys correctly sees `c` (not `j`) for that
    keycode.
18. Press the physical key labelled `c` on QWERTY (which under
    Dvorak produces `j`). Page scrolls down — VimKeys saw `j` from
    the layout-aware resolver.
19. Switch back to ABC / U.S. Confirm the regular `j`/`k`/`h`/`l`
    behaviour is restored within a second (input-source change
    notification refreshes the cache).

## Wrap

20. Quit VimKeys via the menu-bar dropdown.
