# VimKeys v0.1 smoke test

Run on a real Mac with Safari installed. Grant Input Monitoring +
Accessibility when prompted; deny everything else. CI cannot exercise this
checklist — Input Monitoring + Accessibility grants are TCC-gated and
require a real user session.

1. Launch VimKeys.app from /Applications. Menu-bar item appears, no Dock icon.
2. With Safari NOT frontmost, status menu reads "VimKeys — Off".
3. Cmd-Tab to Safari. Status menu now reads "VimKeys — Normal".
4. In Safari with focus on the page (not address bar), press `j`. Page scrolls down ~3 lines.
5. Press `k`. Page scrolls up ~3 lines.
6. Press `h` then `l`. Page scrolls left then right ~3 lines each.
7. Press `d`. Page scrolls down ~15 lines (half-page approximation).
8. Press `u`. Page scrolls up ~15 lines.
9. Press `g` then `g` quickly. Page jumps to top (Cmd+Up effect).
10. Press `Shift+g` (capital G). Page jumps to bottom.
11. Type `5` then `j`. Page scrolls down ~15 lines (5 × 3).
12. Type `12` then `k`. Page scrolls up ~36 lines (12 × 3).
13. Type `g`, wait > 1.5s, then press `g`. Second `g` does nothing — prefix timed out.
14. Type `5gg`. Page jumps to top regardless of count (Vimium convention).
15. Press `Cmd+J` while Safari is frontmost. Cmd+J reaches Safari (Downloads window) — VimKeys does not intercept modified chords.
16. Cmd-Tab away from Safari to any other app. Press `j`. Letter `j` types into the other app — VimKeys is inert when Safari is not frontmost.
17. Cmd-Tab back to Safari. Press `j`. Scroll resumes.
18. In Safari address bar (focused text field), press `j`. The letter `j` types into the field. *Caveat: V-M1 does not yet detect editable focus via AX. The address bar happens to live outside Safari's web-view AX subtree so this just works; in-page text inputs do NOT — a known V-M2 deliverable.*
19. Open VimKeys' Settings window. General tab works; Bindings + Sites tabs both display "Coming in v0.5".
20. Status menu → Quit VimKeys. App exits cleanly.
